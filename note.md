# CRUD 연습 프로젝트 — 학습 노트

할 일 목록(To-do) 앱을 만들며 웹 서비스 개발의 전체 흐름을 익히는 연습 프로젝트.
최종 목표는 이 감을 바탕으로 의료데이터 서비스를 직접 만드는 것.
(이 저장소에는 실제 환자 데이터·민감정보를 절대 올리지 않는다. 공개 데이터만 사용.)

## 큰 그림: 웹 서비스의 3칸
- 프론트엔드 — 브라우저·앱 화면 (사용자가 보는 것)
- 백엔드 — 로직·API (서버에서 도는 처리)
- 데이터베이스 — 데이터 저장소
- 요청→응답 루프: 프론트가 요청 → 백엔드가 처리 → 응답 → 화면에 표시

기술 선택: 백엔드=Flask, DB=SQLite, 프론트=순수 HTML/CSS/JS.
(각각 가장 단순·투명해서 초보가 흐름을 눈으로 볼 수 있는 도구)

## Step 1 — 첫 백엔드 서버 (완료)
목표: 백엔드 칸에 처음 불 켜기.

한 작업:
1. 가상환경: `python -m venv .venv` → `source .venv/bin/activate` → `pip install flask`
   - 프로젝트 전용 격리된 방. 새 터미널/세션마다 activate 다시 필요.
2. `app.py` 작성 (핵심 4줄)
   - `app = Flask(__name__)` : 서버 본체 생성 (Flask=라이브러리 부품, app=내가 지은 이름)
   - `@app.route("/")` : `/` 주소와 바로 아래 함수를 묶어 등록
   - `def home(): return "..."` : 그 요청에 대한 응답 내용
   - `app.run(debug=True)` : 서버 시동 (debug=코드 수정 시 자동 재시작)
3. 실행: `python app.py` → Codespaces PORTS 탭에서 접속 확인

배운 개념:
- home()을 내가 직접 부르지 않는다. 요청이 올 때 Flask가 대신 부른다 (제어의 역전).
- `@`는 아래 함수를 위 도구에 넘기는 문법 (데코레이터).
- return 문자열을 바꾸니 화면도 바뀜 = 내가 서버를 통제한다는 확인.
4. Git 저장: `.gitignore`(.venv/, __pycache__/ 무시) → `git add .` → `git commit` → `git push`
   - 흐름: 작업폴더 →(add) 후보 담기 →(commit) 역사에 박제 →(push) GitHub 백업

## Step 2 — 데이터베이스 붙이기 (완료)
목표: 서버가 데이터를 영구적으로 기억하게 만들기.

왜 필요한가:
- 프로그램의 변수(메모리)는 휘발성 → 서버 재시작하면 사라짐.
- 그래서 데이터를 프로그램 바깥의 파일(=데이터베이스)에 저장한다.

### 2a — 표(table) 만들기 (init_db.py)
- 데이터베이스 = 표들의 모음. 표는 스프레드시트처럼 열(column)+행(row) 구조.
- 우리 todos 표: id(고유번호·자동), content(내용·글자), done(완료여부·0/1)
- SQL = DB에 명령하는 언어. 여기선 `CREATE TABLE`로 표 구조를 만듦.
  - `IF NOT EXISTS`: 이미 있으면 건너뜀 (두 번 돌려도 에러 없음)
  - `PRIMARY KEY AUTOINCREMENT`: 각 행 고유번호를 자동으로 1씩 붙임
  - `NOT NULL`: 비워둘 수 없음 / `DEFAULT 0`: 값 안 주면 0
- 실행하면 todos.db 파일 생성. (SQLite = DB 전체가 파일 하나)

### 2b — 데이터 넣고 읽기 (db_test.py)
- `INSERT INTO ... VALUES (?)`: 새 행 추가 (Create)
- `SELECT * FROM todos`: 데이터 꺼내는 요청 (Read), `*`=모든 열
- `.fetchall()`: SELECT는 '요청'까지, fetchall이 결과를 실제로 받아옴 '수령'
  (찾기와 가져오기를 분리 → 데이터 많을 때 조절 가능)
- `?` 자리표시자: 값을 SQL에 직접 붙이지 않고 분리해서 전달.
  → SQL 인젝션 공격 방지. 사용자 입력 다룰 때 필수 습관.
- 확인: 스크립트 재실행마다 행이 2줄씩 쌓임(2→4→6).
  = 프로그램 꺼져도 데이터가 todos.db 파일에 남아있다는 증거.

### 2c — 서버와 DB 연결 (app.py 수정)
- home() 함수 안에서 connect → SELECT → close 실행.
  라우트 함수 안에 있으니 요청이 올 때마다 매번 DB를 새로 읽음.
- HTML 태그로 화면 구조 표현:
  - `<h1>`=큰 제목, `<ul>`=목록 전체를 감싸는 상자, `<li>`=목록 항목 하나
  - ul과 li는 '감싸는 것 ↔ 담기는 것'의 포함 관계
- f-string `f"...{변수}..."`: 문자열 안 `{}`에 변수 값을 끼워넣는 문법
- `row[0]`=id, `row[1]`=content, `row[2]`=done (파이썬은 0부터 셈)
  → 화면에 뭘 보여줄지는 DB가 아니라 내가 어떤 값을 꺼내 조립하느냐가 결정.

핵심 개념들:
- SQL은 선언형 언어: '어떻게'가 아니라 '무엇을 원하는지'만 씀.
  절차는 쿼리 최적화기(규칙 기반, AI 아님)가 비용 계산해 알아서 처리.
- 데이터 저장 위치: todos.db 파일 → Codespace(클라우드 임시 컴퓨터)의 디스크.
  프로그램 재시작엔 살아남지만 Codespace 삭제 시 사라짐.
- .gitignore에 `*.db` 추가: 데이터 파일은 GitHub에 안 올림.
  이유 ① 실용(데이터는 계속 바뀜, 구조는 init_db.py에 있음)
       ② 보안(실제 데이터는 클라우드에 안 올린다는 원칙)
- 데이터 보관(DB)과 표현(화면)의 분리 = 앞으로 계속 나올 핵심 감각.
