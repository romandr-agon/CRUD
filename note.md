# CRUD 연습 프로젝트 — 학습 노트

할 일 목록(To-do) 앱을 만들며 웹 서비스 개발의 전체 흐름을 익히는 연습 프로젝트.
최종 목표는 이 감을 바탕으로 의료데이터 서비스를 직접 만드는 것.
(이 저장소에는 실제 환자 데이터·민감정보를 절대 올리지 않는다. 공개 데이터만 사용.)

## 큰 그림: 웹 서비스의 3칸
- 프론트엔드 — 브라우저·앱 화면 (사용자가 보는 것)
- 백엔드 — 로직·API (서버에서 도는 처리)
- 데이터베이스 — 데이터 저장소
- 요청→응답 루프: 프론트가 요청 → 백엔드가 처리 → 응답 → 화면에 표시

기술 선택: 백엔드=Flask, DB=SQLite, 프론트=순수 HTML/CSS/JS.
(각각 가장 단순·투명해서 초보가 흐름을 눈으로 볼 수 있는 도구)

## Step 1 — 첫 백엔드 서버 (완료)
목표: 백엔드 칸에 처음 불 켜기.

한 작업:
1. 가상환경: `python -m venv .venv` → `source .venv/bin/activate` → `pip install flask`
   - 프로젝트 전용 격리된 방. 새 터미널/세션마다 activate 다시 필요.
2. `app.py` 작성 (핵심 4줄)
   - `app = Flask(__name__)` : 서버 본체 생성 (Flask=라이브러리 부품, app=내가 지은 이름)
   - `@app.route("/")` : `/` 주소와 바로 아래 함수를 묶어 등록
   - `def home(): return "..."` : 그 요청에 대한 응답 내용
   - `app.run(debug=True)` : 서버 시동 (debug=코드 수정 시 자동 재시작)
3. 실행: `python app.py` → Codespaces PORTS 탭에서 접속 확인

배운 개념:
- home()을 내가 직접 부르지 않는다. 요청이 올 때 Flask가 대신 부른다 (제어의 역전).
- `@`는 아래 함수를 위 도구에 넘기는 문법 (데코레이터).
- return 문자열을 바꾸니 화면도 바뀜 = 내가 서버를 통제한다는 확인.
4. Git 저장: `.gitignore`(.venv/, __pycache__/ 무시) → `git add .` → `git commit` → `git push`
   - 흐름: 작업폴더 →(add) 후보 담기 →(commit) 역사에 박제 →(push) GitHub 백업

## Step 2 — 데이터베이스 붙이기 (완료)
목표: 서버가 데이터를 영구적으로 기억하게 만들기.

왜 필요한가:
- 프로그램의 변수(메모리)는 휘발성 → 서버 재시작하면 사라짐.
- 그래서 데이터를 프로그램 바깥의 파일(=데이터베이스)에 저장한다.

### 2a — 표(table) 만들기 (init_db.py)
- 데이터베이스 = 표들의 모음. 표는 스프레드시트처럼 열(column)+행(row) 구조.
- 우리 todos 표: id(고유번호·자동), content(내용·글자), done(완료여부·0/1)
- SQL = DB에 명령하는 언어. 여기선 `CREATE TABLE`로 표 구조를 만듦.
  - `IF NOT EXISTS`: 이미 있으면 건너뜀 (두 번 돌려도 에러 없음)
  - `PRIMARY KEY AUTOINCREMENT`: 각 행 고유번호를 자동으로 1씩 붙임
  - `NOT NULL`: 비워둘 수 없음 / `DEFAULT 0`: 값 안 주면 0
- 실행하면 todos.db 파일 생성. (SQLite = DB 전체가 파일 하나)

### 2b — 데이터 넣고 읽기 (db_test.py)
- `INSERT INTO ... VALUES (?)`: 새 행 추가 (Create)
- `SELECT * FROM todos`: 데이터 꺼내는 요청 (Read), `*`=모든 열
- `.fetchall()`: SELECT는 '요청'까지, fetchall이 결과를 실제로 받아옴 '수령'
  (찾기와 가져오기를 분리 → 데이터 많을 때 조절 가능)
- `?` 자리표시자: 값을 SQL에 직접 붙이지 않고 분리해서 전달.
  → SQL 인젝션 공격 방지. 사용자 입력 다룰 때 필수 습관.
- 확인: 스크립트 재실행마다 행이 2줄씩 쌓임(2→4→6).
  = 프로그램 꺼져도 데이터가 todos.db 파일에 남아있다는 증거.

### 2c — 서버와 DB 연결 (app.py 수정)
- home() 함수 안에서 connect → SELECT → close 실행.
  라우트 함수 안에 있으니 요청이 올 때마다 매번 DB를 새로 읽음.
- HTML 태그로 화면 구조 표현:
  - `<h1>`=큰 제목, `<ul>`=목록 전체를 감싸는 상자, `<li>`=목록 항목 하나
  - ul과 li는 '감싸는 것 ↔ 담기는 것'의 포함 관계
- f-string `f"...{변수}..."`: 문자열 안 `{}`에 변수 값을 끼워넣는 문법
- `row[0]`=id, `row[1]`=content, `row[2]`=done (파이썬은 0부터 셈)
  → 화면에 뭘 보여줄지는 DB가 아니라 내가 어떤 값을 꺼내 조립하느냐가 결정.

핵심 개념들:
- SQL은 선언형 언어: '어떻게'가 아니라 '무엇을 원하는지'만 씀.
  절차는 쿼리 최적화기(규칙 기반, AI 아님)가 비용 계산해 알아서 처리.
- 데이터 저장 위치: todos.db 파일 → Codespace(클라우드 임시 컴퓨터)의 디스크.
  프로그램 재시작엔 살아남지만 Codespace 삭제 시 사라짐.
- .gitignore에 `*.db` 추가: 데이터 파일은 GitHub에 안 올림.
  이유 ① 실용(데이터는 계속 바뀜, 구조는 init_db.py에 있음)
       ② 보안(실제 데이터는 클라우드에 안 올린다는 원칙)
- 데이터 보관(DB)과 표현(화면)의 분리 = 앞으로 계속 나올 핵심 감각.

## Step 3 — 나머지 CRUD (완료)
목표: 사용자가 화면에서 직접 추가(C)·완료체크(U)·삭제(D)할 수 있게 만들기.
→ 이걸로 CRUD 네 글자 완성: Create(3a), Read(2c), Update(3b), Delete(3c)

### 3a — 할 일 추가 (Create)
- GET과 POST: 요청의 종류(HTTP 메서드).
  GET="보여줘"(데이터 가져오기), POST="이 데이터를 받아 처리해"(만들기/바꾸기)
- form = '형식'이 아니라 '신청서 한 장':
  - `<form>...</form>` = 신청서 종이(입력칸·버튼을 감싸는 울타리)
  - `<input name="content">` = 이름표 붙은 빈칸 (이름표로 서버가 값을 구분)
  - `<button type="submit">` = 제출 도장
  - `action="/add"` = 제출처, `method="post"` = 제출 방법
- 서버 접수 창구: `@app.route("/add", methods=["post"])` + `request.form["content"]`
  → INSERT로 저장 → `redirect("/")`로 목록 화면 복귀
- 라우트는 절차가 아니라 등록부: 서버 시작 시 "주소+방식→함수" 표가 등록되고,
  요청마다 맞는 함수 하나만 실행. 요청과 요청을 잇는 건 브라우저(action, redirect).
- HTTP는 무상태(stateless): 서버는 매 요청을 독립 처리하고 잊는다.
- 브라우저의 정체: ① 사용자 행동을 HTTP 요청으로 번역·발송하는 통신원
  ② 받은 HTML/CSS/JS를 해석해 화면으로 그리는 렌더링 엔진.
  = 프론트엔드 코드의 실행 환경.

### 3b — 완료 체크 (Update)
- 새 문제: 여러 행 중 특정 행을 지정해야 함 → id가 손잡이 역할
- URL 변수: `@app.route("/toggle/<int:todo_id>")` — 주소의 일부를 값으로 받음.
  form의 action에 f-string으로 id를 박아 `/toggle/2`처럼 발송.
- UPDATE SQL: `UPDATE todos SET done = 1 - done WHERE id = ?`
  - WHERE가 생명: 빠지면 모든 행이 바뀜 (UPDATE/DELETE 쓸 땐 WHERE부터 확인)
  - `SET done = 1 - done`: 오른쪽 done은 '현재 값', 왼쪽은 '새로 쓸 자리'.
    DB가 기존 값을 읽고→계산하고→저장. (파이썬 x = x + 1과 같은 구조)
    1-0=1, 1-1=0이라 누를 때마다 0↔1 토글.
- 클릭은 값을 보내는 게 아니라 서버에 새겨진 규칙의 방아쇠를 당기는 것.
  기존 값은 DB가 기억(DEFAULT 0으로 태어남), 규칙은 코드에, 대상 지정만 요청에.
- 조건부 표현식: `mark = "✅" if row[2] == 1 else "⬜"`
  → DB 내부 약속(0/1)을 사람 말로 번역. 번역은 화면 만드는 쪽에서(보관·표현 분리).

### 3c — 삭제 (Delete) + 소프트 삭제
- 하드 삭제: `DELETE FROM todos WHERE id = ?` — 행을 물리적으로 제거.
  SET 없음(값 변경이 아니라 행 제거). WHERE 없으면 표 전체 증발 주의.
- 소프트 삭제(직접 구현함): 행을 없애는 대신 표시만 하는 방식
  ① 표에 열 추가: `deleted INTEGER NOT NULL DEFAULT 0`
  ② 삭제 라우트 SQL 교체: `UPDATE todos SET deleted = 1 WHERE id = ?`
  ③ 목록 SELECT에 필터: `SELECT * FROM todos WHERE deleted = 0`
  → 화면에선 사라지지만 DB에는 보존. 복구는 `SET deleted = 0` 한 줄.
  → 의료 데이터처럼 기록 보존이 필요한 곳에서 사용. 단 표가 비대해지고
    모든 SELECT에 필터를 빼먹으면 안 되는 부담. 진짜 지워야 하는 의무(잊힐 권리)도
    있으므로 하드/소프트는 요구사항에 따른 설계 선택.

### 그 외 배운 것
- 표 구조 변경: `CREATE TABLE IF NOT EXISTS`는 기존 표를 안 고침.
  방법 A) `rm todos.db` 후 재생성(연습용) B) `ALTER TABLE ... ADD COLUMN`(실무, 마이그레이션)
- SQL 문법: 열 정의 사이엔 쉼표 필수(마지막 항목 뒤에 추가할 때 빼먹기 쉬움)
- `rm` = 리눅스 파일 삭제 명령. 휴지통 없이 즉시 영구 삭제라 신중히.
  (파일시스템의 휴지통과 DB의 soft delete는 같은 문제의식의 해법)
- DB는 서버의 소유물이 아닌 공용 창고: Flask도 sqlite3 터미널 도구도 같은 DB에
  접근하는 별개의 손님. 관리자 직접 접근은 앱의 규칙을 우회하므로 강력하지만 위험.
  (실무·의료에선 직접 접근을 통제하고 감사 기록을 남김)
- AUTOINCREMENT는 번호 재사용 안 함(은행 대기표): id는 순번이 아니라 영구 신분증.
  재사용하면 지워진 id를 새 행이 물려받아 엉뚱한 행을 조작하는 사고 발생.
  id에 구멍이 있는 건 정상(뭔가 지워진 흔적).

## 막간 — 용어·구조 대정리 (Step 4 진입 전)
코드를 다시 읽다가 나온 근본 질문들. 카테고리 지도가 잡힌 날.

### 이름의 네 세계 (content가 4번 나와도 안 헷갈리는 이유)
- 같은 철자라도 소속이 다르다: ① form 이름표("content", HTML name과 짝)
  ② 파이썬 변수(자유, 자기들끼리 짝) ③ DB 열 이름(표 구조에 고정) ④ ②와 같은 변수
- 값의 릴레이: 입력칸 → 이름표 달고 발송 → 변수에 담김 → SQL ?에 끼워짐 → 열에 저장
- 전부 content로 통일한 건 기술적 필연이 아니라 읽는 사람을 위한 관례
- 컴퓨터는 문맥으로 추측하지 않는다: 따옴표 유무, 등호 좌우, 해석 주체에 따라
  찾아갈 서랍(이름공간)이 문법적으로 정해져 있음. 로컬 변수는 함수 끝나면 소멸
  → 요청 간 데이터가 안 섞임 (무상태성의 파이썬 레벨 구현)

### conn.close()를 하는 이유
- connect/execute/fetchall은 '정의'가 아니라 즉시 실행. fetchall 끝난 순간
  데이터는 이미 파이썬 리스트로 복사됨(사본) → 이후 for문은 DB 불필요
- 연결은 자원 점유 + 잠금 위험 → 필요한 최소 구간만 열고 즉시 닫는다

### 언어 카테고리 지도
- 언어(표기법, 서로 대등): Python·JS(절차 지시) / SQL(선언형) / HTML(마크업) / CSS(스타일)
- 해석자(언어를 읽는 프로그램): Python↔인터프리터, HTML·CSS·JS↔브라우저, SQL↔SQLite
- 라이브러리(언어로 쓰인 부품): Flask, sqlite3 모듈 — 언어에 소속됨
- 규약: HTTP — 설치하는 물건이 아니라 브라우저·서버가 함께 따르는 편지 양식
  (GET/POST, Content-Type 꼬리표, redirect, 무상태성이 전부 이 규약의 조항)
- 새 용어 분류법: "언어인가, 라이브러리인가, 해석자인가, 규약인가"

### 헷갈렸던 대칭 바로잡기
- HTML은 실행하는 언어가 아니라 해석되는 마크업: html= 변수명은 무의미(frontend도 됨),
  따옴표 안 내용이 브라우저에 도착해 해석되는 순간 비로소 HTML로 작동
- 서버 ≠ 해석자: 서버=요청을 기다리는 프로그램 / 해석자=글을 읽는 프로그램.
  언어는 서버가 필요 없다(해석자만 필요). DB는 배치 선택(SQLite=내장형, PostgreSQL=서버형)
- SQLite vs Flask는 비대칭: SQLite는 SQL을 '읽는' 해석자(C로 만들어짐),
  Flask는 Python으로 '쓰인' 라이브러리. "읽는 놈인가 쓰인 놈인가"로 구분
- SQLite 해석자는 따로 설치 안 함: 파이썬에 내장돼 딸려옴(임베디드 설계).
  터미널의 sqlite3 명령은 같은 엔진을 품은 별도 조회 도구(같은 DB 파일을 보는 두 손님)
- 해석자의 계보: 파이썬 인터프리터는 C로, C 컴파일러는 이전 C로, 최초엔 어셈블리·기계어로
  — 부트스트래핑. 언어는 사다리 구조
- 브라우저에 파이썬 엔진을 안 넣는 진짜 이유: 크기가 아니라 ①JS 표준 생태계
  ②신뢰 경계 — 백엔드 코드(DB 접근)가 사용자 손에 넘어가면 안 됨.
  서버=내가 통제하는 영역, 브라우저=남이 통제하는 영역 (의료 서비스 보안의 핵심 경계)
- 프레임워크가 대신 해주는 것: HTTP 포장(상태코드·헤더), 요청 해석(request.form),
  등록부 대조, URL 변수 변환… — 뭘 대신해주는지 알고 쓰는 것과 모르고 쓰는 것의 차이
